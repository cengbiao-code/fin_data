# HK PDF Text Encoding Diagnostics Design

Date: 2026-06-13

## Goal

Add a diagnostic command and experimental decoder for Hong Kong financial-report
PDFs whose extracted text is unreadable or unusable for table classification.

The canonical domain term is **港股 PDF 文本编码诊断**. This work diagnoses text
encoding and CMap failures; it is not visual font recognition and it is not OCR.

## Problem

Some Hong Kong Chinese financial reports generated by Adobe InDesign use
Traditional Chinese CID fonts and Big5-family CMaps such as `ETen-B5-H`.
When a PDF lacks a usable `/ToUnicode` map, libraries such as pdfplumber,
pdfminer, or PyMuPDF may return mojibake, incorrect Unicode, or CID-like text.
The downstream classifier then cannot match statement titles, so the three main
financial statements are not detected.

## Scope

Implement option B from the design discussion:

1. A PDF font and text-layer diagnostic command.
2. An experimental content-stream decoding prototype using existing
   `pdfminer.six` capabilities.

This work does not automatically repair trusted extraction output.

## Non-Goals

- Do not write candidate decoded text to `raw_tables`, `raw_cells`, trusted
  facts, or any audit database table.
- Do not change the trusted publishing rules.
- Do not introduce OCR.
- Do not solve table grid reconstruction or cell coordinates in this phase.
- Do not add `pypdf` or other new dependencies unless a later design changes
  that decision.

## Command

Add:

```powershell
fin-report inspect-pdf-fonts report.pdf --pages 1-20
fin-report inspect-pdf-fonts report.pdf --pages 1-20 --json
```

Default output is human-readable text. `--json` emits a stable machine-readable
report suitable for tests and future fixtures.

## Classification

Each report receives one primary diagnostic classification:

- `healthy_text_layer`: Existing text extraction appears usable. If statement
  tables still fail, the likely cause is outside text encoding.
- `mojibake_repairable`: Existing extracted text is damaged, but
  `repair_pdf_text()` materially improves it.
- `missing_tounicode_decode_candidate`: The PDF uses HK Big5 / Adobe-CNS1-like
  font resources and missing or poor `/ToUnicode` mappings; the experimental
  decoder produces a better candidate text sample.
- `unrecoverable_or_ocr_required`: The text cannot be recovered reliably from
  the available text layer or content stream.

Layout-specific failures may be reported as notes, but they are not top-level
classifications in this feature.

## Components

### `pdf_font_inspector.py`

Reads PDF metadata and page/font information, producing:

- document producer when available;
- pages inspected;
- font resource name;
- base font;
- encoding or CMap name when discoverable;
- whether `/ToUnicode` exists;
- whether the font looks like HK Big5 / Adobe-CNS1;
- extracted text sample;
- repaired text sample;
- text-quality scores.

### `hk_content_decoder.py`

Experimental decoder based on existing `pdfminer.six` APIs and bundled CMap
resources. It attempts to recover candidate text from page content streams by:

- inspecting text-showing operations such as `Tj` and `TJ`;
- tracking the currently selected font where practical;
- trying Big5-family decoding strategies for HK Big5 / Adobe-CNS1-like fonts;
- scoring candidates by Traditional Chinese CJK ratio, Hong Kong statement
  keywords, mojibake indicators, and improvement over the existing text layer.

The decoder returns candidate decoded text only. It does not produce raw table
evidence.

## Report Shape

JSON output should follow this conceptual shape:

```json
{
  "pdf_path": "...",
  "producer": "Adobe InDesign ...",
  "pages_inspected": [1, 2, 3],
  "classification": "missing_tounicode_decode_candidate",
  "summary": {
    "hk_cmap_font_count": 1,
    "fonts_missing_tounicode": 1,
    "candidate_decoder_count": 1
  },
  "pages": [
    {
      "page_number": 1,
      "classification": "missing_tounicode_decode_candidate",
      "font_findings": [],
      "text_layer_sample": "...",
      "repaired_sample": "...",
      "decoded_candidates": [
        {
          "strategy": "big5hkscs",
          "score": 0.91,
          "preview": "綜合財務狀況表...",
          "reason": "HK statement keywords improved"
        }
      ],
      "notes": []
    }
  ]
}
```

The first version does not guarantee bbox, row, column, or table structure.

## Testing

Add focused tests for:

- font findings with and without `/ToUnicode`;
- HK Big5 / Adobe-CNS1-like font detection;
- mojibake repair classification;
- experimental decoder candidate scoring using small fake or synthetic content
  objects;
- CLI text output;
- CLI `--json` output.

Real Tencent PDF verification should be handled as a manual smoke test unless a
shareable fixture is added later.

## Acceptance Criteria

- `fin-report inspect-pdf-fonts` runs on a local PDF path.
- The command can limit inspected pages.
- The command prints a clear primary classification and recommended next step.
- `--json` produces stable structured output.
- Candidate decoded text is never persisted into SQLite or DuckDB.
- Existing full test suite passes.
